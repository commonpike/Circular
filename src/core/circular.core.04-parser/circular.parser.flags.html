When {{expressions}} are processed several steps are taken, which can be controlled by adding flags after a pipe, like this: <div class="{{color:@brush.foreground|pi}}">.
These are the flags available:
  • p:parse
    Perform string operations on the expression to resolve all references to #this,$this,# and @.
  • e:evaluate
    Evaluate the resulting expression - much like javascripts eval.
  • i:insert
    Re-insert the result of the previous operations back into the dom attribute where it came from.
  • w:watch
    Watch all variables used in the expression for changes
The order of the flags is irrelevant. When no flags are passed, all flags are assumed on, so {{flush()}} equals {{flush()|wipe}}.

This may be useful, for example, if you don't want the expression to be evaluated when parsed, like in event handlers:
<a onclick="{{alert('#name')|wip}}">what's my name</a>
If we would add the flag 'e' in that code, circular would alert #name (or its value) while parsing the expression, not when clicking on the link.

Even if the 'i' flag is not used, effectively leaving the original expression in the attribute, the parsed, evaluated and watched result of the expression is still available in the @registry, as 'attribute metadata'.
